# Gemini CLI 엔지니어링 가이드 (GEMINI.md)

이 파일은 Gemini CLI가 프로젝트의 맥락을 이해하고 고품질의 코드를 자율적으로 생산하기 위한 **강제 지침서**입니다.

## 0. 지침 준수 선언 (MANDATORY)
- 모든 작업 시작 전 이 파일을 재독하고, 해당되는 규칙을 **체크리스트**로 작성하여 출력합니다.
- 최종 보고 시 체크리스트 준수 여부를 명시합니다. 지침 위반 시 작업은 실패로 간주됩니다.

## 1. 기본 원칙 (Core Principles)
- **언어**: 모든 상호작용 및 주석, 문서는 **한국어**로 수행합니다.
- **범위 준수**: 현재 요청과 직접 관련 없는 코드는 수정하지 않는 **최소 수정 원칙**을 고수합니다.
- **맥락 우선**: 프로젝트 스택(Next.js 15, TS, Tailwind, Supabase, Drizzle)에 최적화된 코드를 작성합니다.
- **논리-텍스트 정합성**: 비즈니스 로직(DB 스키마, 서버 액션 등) 변경 시, 해당 기능을 설명하는 모든 UI 텍스트(안내문, 모달 메시지, 플레이스홀더 등)를 반드시 전수 조사하여 일치시킵니다.

## 2. 개발 사이클 (Plan-Act-Validate)
1. **점검 및 계획 (Plan)**: 
   - 환경 변수 및 DB 스키마 존재 여부를 최우선으로 검증합니다.
   - 수정 대상과 의존성을 분석하여 전략을 세웁니다.
   - **텍스트 영향도 분석**: 로직 수정이 사용자 안내 문구에 영향을 주는지 사전에 파악합니다.
2. **구현 (Act)**: 
   - **타입 안전성**: `any` 사용을 엄격히 금지하며, 인터페이스보다 타입을 선호합니다.
   - **컴포넌트 구조**: UI는 `app/components`에 위치시키며, Server Component를 우선합니다.
   - **네이밍**: 파일은 `kebab-case`, 컴포넌트는 `PascalCase`를 사용합니다.
   - **문구 동기화**: 로직 수정과 동시에 관련된 UI 텍스트를 함께 수정합니다.
3. **검증 (Validate)**: 
   - **단위 테스트**: 기능 구현 후 반드시 `vitest`를 통해 비즈니스 로직을 검증합니다.
   - **자동 검증**: 보고 전 반드시 `npm run verify`를 실행하여 Lint, TSC, Test 통과를 확인합니다.
   - **최종 정합성 확인**: 수정된 기능의 실제 동작과 UI 상의 설명이 일치하는지 `grep` 검색 등을 통해 다시 한번 확인합니다.

## 3. 기술적 모범 사례 (Technical Best Practices)

### 3.1. 반응형 UI 및 상태 관리
- **낙관적 업데이트 (Optimistic UI)**: `useOptimistic` 업데이트 함수는 반드시 `startTransition` 콜백 내부나 서버 액션 실행문 내에서 호출하여 브라우저 경고를 방지합니다.
- **에러 핸들링**: 서버 액션 실패 시(`try-catch`) 사용자에게 명확한 피드백(alert 등)을 제공하고 기존 상태를 유지합니다.

### 3.2. 하이드레이션 안전성 (Hydration Safety)
- **날짜/시간 표시**: `toLocaleString()` 등 로케일 의존 API 사용 시, 서버/클라이언트 텍스트 불일치를 막기 위해 해당 요소에 `suppressHydrationWarning`을 적용합니다.

### 3.3. Next.js 15 최적화
- **비동기 정합성**: `params`, `searchParams`, `cookies`, `headers` 등은 반드시 `await` 처리합니다.
- **서버 액션 리다이렉트**: 클라이언트의 `try-catch`에서 `NEXT_REDIRECT` 에러가 발생하는 것을 방지하기 위해, 리다이렉트 로직은 가급적 클라이언트(`router.push`)에서 제어합니다.

## 4. 데이터베이스 및 보안 (DB & Security)
- **스키마 실재 검증**: DDL 작업 후 반드시 `mcp_supabase_list_tables`로 반영을 확인합니다.
- **소유권 검증 (Authorization)**: 모든 데이터 변경(CUD) 요청 시 서버 사이드에서 `authorId`와 현재 `user.id`가 일치하는지 엄격히 검사합니다.
- **개인정보 보호**: 공개 화면에서는 이메일 대신 **익명 닉네임**만을 활용합니다.
- **관리자 권한**: 고권한 작업(계정 삭제 등)은 반드시 **Edge Functions**를 통해 격리된 환경에서 수행합니다.

## 5. UX 완성도 및 예외 처리 (UX & Error Handling)
- **무반응 방지**: 모든 클릭 가능한 요소(버튼, 링크 등)는 데이터가 없거나 작업이 불가능한 상태일 때 반드시 사용자에게 피드백을 제공해야 합니다. (예: `cursor-default` 스타일 적용 및 알림 메시지 출력)
- **데이터 부재 대응**: 프로필 정보 부재(탈퇴 사용자), 게시글 삭제 등 데이터가 `null`인 상황을 고려한 방어적 코드를 작성하고 UI적으로 이를 명확히 표현합니다.
- **로딩 및 에러 상태**: 비동기 작업 시 사용자가 진행 상황을 알 수 있도록 스켈레톤 UI나 로딩 스피너를 제공하고, 에러 발생 시 시스템 중심의 메시지가 아닌 사용자 중심의 친절한 안내를 제공합니다.
