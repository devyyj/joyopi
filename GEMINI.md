# Gemini CLI 엔지니어링 가이드 (GEMINI.md)

이 파일은 Gemini CLI가 프로젝트의 맥락을 이해하고 고품질의 코드를 자율적으로 생산하기 위한 **강제 지침서**입니다.

## 0. 지침 준수 선언 (MANDATORY)
- 모든 작업 시작 전 이 파일을 재독하고, 해당되는 규칙을 **체크리스트**로 작성하여 출력합니다.
- 최종 보고 시 체크리스트 준수 여부를 명시합니다. 지침 위반 시 작업은 실패로 간주됩니다.

## 1. 기본 원칙 (Core Principles)
- **언어**: 모든 상호작용 및 주석, 문서는 **한국어**로 수행합니다.
- **범위 준수**: 현재 요청과 직접 관련 없는 코드는 수정하지 않는 **최소 수정 원칙**을 고수합니다.
- **맥락 우선**: 프로젝트 스택(Next.js 15, TS, Tailwind, Supabase, Drizzle)에 최적화된 코드를 작성합니다.
- **논리-텍스트 정합성**: 비즈니스 로직(DB 스키마, 서버 액션 등) 변경 시, 해당 기능을 설명하는 모든 UI 텍스트(안내문, 모달 메시지, 플레이스홀더 등)를 반드시 전수 조사하여 일치시킵니다.

## 2. 개발 사이클 (TDD: Test-Driven Development) - MANDATORY
모든 기능 개발과 버그 수정은 반드시 **Red-Green-Refactor** 사이클을 따릅니다.

1.  **점검 및 설계 (Plan)**: 
    - DB 스키마 및 인프라 의존성(Storage 버킷 등)을 분석합니다.
    - **실패하는 테스트 케이스 작성 (Red)**: 구현할 기능이나 수정할 버그를 검증하는 테스트 코드를 **먼저** 작성합니다. 이미지 업로드 등 외부 서비스 의존성은 모킹하되, 성공/실패 시나리오를 전수 포함합니다.
2.  **구현 및 성공 (Green)**: 
    - 테스트를 통과할 만큼의 **최소한의 코드**를 작성합니다.
    - **자동 수정**: 코드 수정 시 `eslint --fix`가 자동으로 실행됩니다.
3.  **리팩터링 및 검증 (Refactor/Validate)**: 
    - 코드 품질을 개선하고 중복을 제거합니다. 리팩터링 후에도 테스트가 통과하는지 확인합니다.
    - **인프라 검증**: Storage 버킷 등 외부 설정이 필요한 경우, `mcp_supabase_execute_sql` 등을 통해 실제 환경에 버킷과 정책이 존재하는지 직접 검증합니다.
    - **최종 정합성 확인**: `npm run verify`를 실행하여 전체 시스템의 안전성을 확인합니다.

## 3. 기술적 모범 사례 (Technical Best Practices)

### 3.1. 테스트 원칙
- **기능별 전수 테스트**: 모든 서버 액션은 성공 시나리오뿐만 아니라 입력값 유효성 실패, 권한 부족, 외부 서비스(Storage) 에러 등 예외 시나리오를 최소 3개 이상 포함해야 합니다.
- **이미지/파일 처리 테스트**: 파일 업로드 로직이 포함된 경우 반드시 `mockStorage`를 정교하게 구성하여 업로드 실패 및 URL 생성 실패 상황을 테스트합니다.
- **TDD 미준수 시 작업 실패**: 테스트 코드 없이 로직만 수정된 PR이나 작업은 지침 위반으로 간주합니다.

### 3.1. 반응형 UI 및 상태 관리
- **낙관적 업데이트 (Optimistic UI)**: `useOptimistic` 업데이트 함수는 반드시 `startTransition` 콜백 내부나 서버 액션 실행문 내에서 호출하여 브라우저 경고를 방지합니다.
- **에러 핸들링**: 서버 액션 실패 시(`try-catch`) 사용자에게 명확한 피드백(alert 등)을 제공하고 기존 상태를 유지합니다.

### 3.2. 하이드레이션 안전성 (Hydration Safety)
- **날짜/시간 표시**: `toLocaleString()` 등 로케일 의존 API 사용 시, 서버/클라이언트 텍스트 불일치를 막기 위해 해당 요소에 `suppressHydrationWarning`을 적용합니다.

### 3.3. Next.js 15 최적화
- **비동기 정합성**: `params`, `searchParams`, `cookies`, `headers` 등은 반드시 `await` 처리합니다.
- **서버 액션 리다이렉트**: 클라이언트의 `try-catch`에서 `NEXT_REDIRECT` 에러가 발생하는 것을 방지하기 위해, 리다이렉트 로직은 가급적 클라이언트(`router.push`)에서 제어합니다.

## 4. 자가 개선 및 재발 방지 (Self-Correction) - MANDATORY
에러나 버그가 발견되면 단순히 코드만 수정하지 않고, **시스템적으로 재발을 막기 위한 조치**를 자동 수행합니다.

1.  **사후 분석 (Post-Mortem)**: 버그 수정 후 "왜 테스트나 검증 단계에서 걸러지지 않았는가?"를 자문하고 근본 원인을 기록합니다.
2.  **프로세스 업데이트**:
    -   유사한 실수가 반복되지 않도록 본 지침서(`GEMINI.md`)의 '기술 체크리스트'를 즉시 보완합니다.
    -   필요시 `.gemini/hooks`나 `verify.ps1`에 자동화된 검사 로직을 추가합니다.
3.  **지식 축적**: `save_memory`를 통해 프로젝트 특유의 함정(Gotchas)을 기록하고 매 작업 시작 시 복기합니다.

## 5. 기술 체크리스트 (Technical Checklist)

### 5.1. Next.js 15 & Image (CRITICAL)
- [ ] **Image 레이아웃**: `fill` 속성 사용 시 부모 요소에 반드시 `relative`, `absolute`, 또는 `fixed` 포지션이 있는지 확인합니다.
- [ ] **Image 높이 보장**: `fill` 이미지의 부모 요소는 반드시 명시적인 `height`나 `aspect-ratio`를 가져야 합니다. (0px 렌더링 방지)
- [ ] **성능 최적화**: 모든 `Image` 컴포넌트에 적절한 `sizes` 프롭을 적용합니다. (LCP 경고 방지)
- [ ] **도메인 허용**: 새로운 외부 이미지 소스 사용 시 `next.config.ts`의 `remotePatterns`를 업데이트합니다.

### 5.2. 데이터 동기화 & 낙관적 업데이트
- [ ] **상태 동기화**: `revalidatePath` 사용 시 클라이언트 컴포넌트가 최신 프롭을 상태에 동기화하는지 확인합니다. (렌더링 도중 상태 갱신 패턴 권장)
- [ ] **롤백 방지**: 서버 응답 후 클라이언트 상태가 서버 데이터와 일치하도록 보장하여 `useOptimistic`의 롤백 현상을 방지합니다.

### 5.3. 인프라 의존성
- [ ] **스토리지/DB**: 코드가 참조하는 모든 Supabase 버킷, RLS 정책, DB 스키마가 실제 환경에 존재하는지 `mcp` 도구로 사전 검증합니다.

## 6. 학습된 프로젝트 지침 (Learned Lessons)

### 6.1. 데이터베이스 연결 (Next.js HMR 대응)
- Next.js 개발 환경에서 DB 연결 객체가 중첩 생성되어 'max client connections reached' 에러가 발생하는 것을 방지하기 위해, `db/index.ts` 등에서 연결 인스턴스를 `globalThis`에 캐싱하여 재사용해야 합니다.

### 6.2. Next.js 15 이미지 최적화
- `Image` 컴포넌트의 `fill` 속성 사용 시 부모 요소에 반드시 `relative` 포지션이 있어야 하며, 명시적인 높이(`height`)나 `aspect-ratio`를 가져야 이미지가 0px로 렌더링되는 것을 방지할 수 있습니다.
- 성능 경고 방지를 위해 모든 이미지에 적절한 `sizes` 프롭을 반드시 지정합니다.

### 6.3. 데이터 동기화 및 롤백 방지
- `revalidatePath` 호출 시 서버 데이터는 갱신되지만, `PostList`와 같이 내부 상태(`useState`)를 가진 클라이언트 컴포넌트는 새로운 프롭을 상태에 명시적으로 동기화(렌더링 도중 갱신 패턴 등)해야 낙관적 업데이트의 롤백 현상을 방지할 수 있습니다.

### 6.4. 메모리 및 설정 스코프 (CRITICAL)
- **절대 Global Scope로 메모리를 저장하지 않습니다.** 모든 설정과 학습 데이터는 오직 Project Scope로만 관리하며, 글로벌 설정이 발견될 경우 즉시 프로젝트 지침으로 이전합니다.
