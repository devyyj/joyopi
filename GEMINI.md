# Gemini CLI 설정 및 사용 가이드 (GEMINI.md)

이 파일은 Gemini CLI가 프로젝트의 맥락을 정확히 이해하고 사용자의 의도에 맞는 고품질의 결과를 내놓도록 돕는 지침서입니다.

## 0. 지침 준수 강제 규칙 (MANDATORY)
**이 파일의 모든 지침은 선택 사항이 아닌 강제 명령입니다.**
- 작업을 시작하기 전, 반드시 이 파일을 다시 읽고 이번 작업에 해당되는 규칙을 나열(Checklist)한 후 진행합니다.
- 최종 보고 시, 위 체크리스트의 각 항목을 어떻게 준수했는지 명시해야 합니다.
- 지침을 위반할 경우, 작업은 실패로 간주하며 스스로 원인을 분석하여 다시 수행해야 합니다.

## 1. 기본 원칙 (General Principles)
- **언어 설정 (Language)**: 모든 상호작용은 반드시 **한국어(한글)**로 수행합니다.
- **프로젝트 독립성**: 이 프로젝트는 전역 설정에 의존하지 않고, `.gemini/` 및 `GEMINI.md`의 지침을 최우선으로 하여 자체적으로 관리됩니다.
- **구체성 우선**: 모호한 요청보다 구체적인 제약 조건을 선호합니다.
- **맥락 우선**: 프로젝트의 기술 스택, 아키텍처, 코딩 스타일을 명시하여 일관성을 유지합니다.
- **부정적 제약 조건**: "하지 말아야 할 것"을 명시하는 것이 모호한 긍정 지침보다 효과적입니다.

## 2. 권장되는 GEMINI.md 구조

### 프로젝트 정보 및 기술 스택
- **Framework**: Next.js (App Router)
- **Language**: TypeScript
- **Styling**: Tailwind CSS (선호됨)
- **State Management**: (프로젝트에 맞는 라이브러리 명시)

### 코딩 규칙 (Conventions)
- **디자인 테마 (Design Theme)**: 전체적인 UI/UX는 **GitHub (Primer Design System)** 스타일을 지향합니다.
  - **색상**: GitHub 고유의 파란색, 무채색 배경, 실선 테두리를 사용합니다.
  - **컴포넌트**: `rounded-md`(약 6px) 수준의 모서리 곡률을 사용하며, 과도한 그라데이션이나 화려한 그림자 대신 정갈한 보더와 대비를 활용합니다.
  - **폰트**: 시스템 폰트 스택을 우선하며, 가독성을 최우선으로 합니다.
- **함수 정의**: 화살표 함수(`const Func = () => ...`)를 기본으로 사용합니다.
- **컴포넌트 구조**: UI 컴포넌트는 `app/components`에 위치시키며, 가급적 Server Component를 우선합니다.
- **타입 정의**: `any` 사용을 엄격히 금지하며, 인터페이스보다는 타입을 선호합니다.
- **네이밍**: 파일명은 `kebab-case`, 컴포넌트명은 `PascalCase`를 사용합니다.

### 자동화된 개발 사이클 (Autonomous Development Cycle)
명령을 받으면 Gemini CLI는 다음의 **순환적이고 자율적인 프로세스**를 반드시 준수해야 합니다.

0. **범위 준수 (Scope Management)**:
   - **현재 요청에만 집중**: 사용자의 현재 지시 사항과 직접적으로 관련이 없는 파일이나 기능을 수정하지 마십시오.
   - **이전 컨텍스트 격리**: 이전 대화에서 언급된 요구 사항이라도, 현재 작업 범위에 포함되지 않는다면 무분별하게 반영하지 마십시오. 필요 시 사용자에게 범위를 확인하십시오.
   - **최소 수정 원칙**: 목적 달성을 위한 최소한의 변경만을 수행하며, 부수적인 '청소'나 '개선' 작업은 별도의 제안으로 남기십시오.

1. **점검 및 계획 (Plan)**: 
   - 사용자의 명령과 요구 조건을 철저히 분석하고, 수정이 필요한 파일과 연관된 코드(의존성)를 탐색합니다.
   - **가설을 세우기 전, 반드시 관련 환경 변수가 설정되어 있는지(`.env` 확인 등) 실재를 먼저 검증하십시오.**
   - **`GEMINI.md`를 검토하여 이번 작업에 적용할 "준수 체크리스트"를 작성하여 출력합니다.**
   - 작업을 시작하기 전에 어떠한 방식으로 구현하고 **유닛 테스트를 통해 어떻게 검증할 것인지** 전략을 세웁니다.
2. **개발 및 구현 (Execute)**: 
   - 계획한 내용에 따라 코드를 작성하거나 수정합니다.
   - **기능 구현이 완료되면 해당 기능을 검증할 수 있는 유닛 테스트를 반드시 작성합니다.**
3. **테스트 및 검증 (Validate & MANDATORY)**: 
   - 코드를 수정한 후, **반드시 `npm run verify` 명령을 실행하여 자동 검증을 수행**합니다.
   - **인프라 통합 검증 (Integration Critical)**: Edge Functions나 외부 API 연동 시, 단위 테스트(Mock)의 통과가 실제 환경의 성공을 보장하지 않음을 명심하십시오.
   - **개인정보 보호 원칙 (Privacy-First)**:
     - **최소 수집**: 구글 로그인 등 외부 인증 시 제공되는 이메일, 이름(`full_name`), 프로필 이미지(`avatar_url`) 등을 절대 DB에 저장하거나 서버 로그로 남기지 마십시오.
     - **식별자 사용**: 오직 사용자의 UUID(`auth.users.id`)만 식별자로 사용하며, 서비스 내에서는 무작위로 생성된 익명 닉네임만을 활용하십시오.
     - **데이터 격리**: `auth.users`에 저장된 개인정보와 `public` 스키마의 활동 데이터가 물리적으로 연결되지 않도록 설계하십시오. (프로필 테이블에 이메일 저장 금지)
   - **사용자에게 보고하기 전, 검증 결과가 모두 통과(Success)되었음을 반드시 확인하고 보고서에 명시해야 합니다.**
   - 필요 시 `npm run build`를 추가로 실행하여 전체 빌드 성공 여부를 확인합니다.
   - 결과물이 사용자가 제시한 초기 조건을 모두 만족하는지 스스로 검토합니다.
4. **결과 검토 및 자가 수정 (Review & Self-Correct)**:
   - 만약 테스트나 빌드 과정에서 **에러가 발생하거나, 요구 조건을 충족하지 못한 경우**, 사용자에게 즉시 실패를 보고하는 대신 **스스로 원인을 분석하고 1단계(Plan)부터 다시 시작하여 오류를 수정**합니다.
   - 이 과정을 **모든 에러가 해결되고 조건이 완벽히 충족될 때까지** 반복합니다.
5. **자가 점검 및 수정 프로세스 (Self-Correction Strategy)**:
   - 최종 보고 전, 반드시 다음 체크리스트를 수행하여 자가 점검을 완료합니다.
   - **사용하지 않는 코드 및 파일 삭제**: 개발 과정에서 더 이상 사용되지 않게 된 코드(함수, 변수, 임포트)나 파일을 점검하고 반드시 삭제합니다.
   - **빌드 및 타입 검증**: `npm run verify` (ESLint + tsc) 실행 및 성공 확인.
   - **유닛 테스트 검증**: 작성된 모든 유닛 테스트 통과 여부 확인.
   - **비동기 정합성**: Next.js 15+ 환경에서 `params`, `searchParams`, `cookies`, `headers`가 올바르게 `await` 처리되었는지 전수 조사.
   - **환경 변수 실무 점검**: 핵심 환경 변수(DATABASE_URL, SUPABASE_URL, NEXT_PUBLIC_SITE_URL 등)가 실제 실행 환경에 설정되어 있는지 확인하고, 누락 시 발생할 부작용을 사전에 차단하십시오. (단순 폴백 유무만 확인하는 것에 그치지 말 것)
   - **에러 핸들링**: DB 연결 실패 등 외부 서비스 장애 상황에서도 어플리케이션이 크래시되지 않도록 `try-catch` 또는 에러 경계(`error.tsx`)가 있는지 점검.
6. **DB 무결성 및 권한 자가 점검 (Database Integrity & RLS Strategy)**:
   - DML(INSERT/UPDATE/DELETE) 작업이 포함된 코드를 작성할 때는 반드시 다음을 점검합니다.
   - **스키마 동기화 확인**: 로컬의 `db/schema.ts` 정의가 실제 DB 테이블 구조와 100% 일치하는지 `mcp_supabase_list_tables`로 확인.
   - **RLS 정책 검토**: `INSERT` 작업 전, 해당 테이블에 `authenticated` 사용자를 위한 권한 정책이 수립되어 있는지 확인. (Supabase는 기본적으로 모든 INSERT를 차단함)
   - **데이터 타입 검증**: UUID, JSONB 등 특수 타입 사용 시 전달되는 값이 규격에 맞는지 타입 캐스팅 여부 확인.
   - **FK 제약 조건**: 참조하는 ID(예: user.id)가 실제 존재하는 데이터인지, 혹은 `auth.users`를 올바르게 참조하는지 점검.
   - **회원 탈퇴(Withdrawal) 및 관리자 권한 작업 원칙**: 
     - **Service Role Key 사용 금지**: `SUPABASE_SERVICE_ROLE_KEY`를 Next.js 애플리케이션의 환경 변수(`.env`)로 절대 등록하지 마십시오.
     - **격리된 실행 환경(Edge Functions)**: 계정 삭제, 권한 변경 등 관리자 권한이 필요한 작업은 반드시 Supabase Edge Functions를 통해 수행하십시오.
     - **최소 권한 원칙**: 애플리케이션은 사용자의 JWT만 전달하며, 실제 고권한 작업은 인프라 수준에서 격리된 함수가 담당합니다.
     - **직접 삭제 금지**: `auth.users` 테이블을 직접 조작하지 마십시오.
     - **CASCADE 활용**: DB 레벨의 `ON DELETE CASCADE`를 통해 데이터 무결성을 보장하십시오.
7. **동적 URL 관리 (Dynamic URL Strategy)**:
   - **현재 호스트 직접 참조**: 고정된 도메인이나 복잡한 유틸리티 대신, 현재 실행 환경의 표준 방식을 사용하여 호스트를 참조하십시오.
   - **Route Handlers**: `request.url`로부터 `new URL(request.url).origin`을 추출하여 사용하십시오.
   - **Server Actions**: `next/headers`의 `headers()`를 사용하여 현재 접속 중인 `host`를 동적으로 감지하십시오. 로컬 환경(`localhost`)인 경우 `http`, 그 외 배포 환경인 경우 `https` 프로토콜을 사용하십시오.
   - **외부 서비스 설정 동기화**: Supabase, Google Console 등 외부 서비스의 리다이렉트 허용 목록(Whitelist)에 현재 배포 환경의 URL(`https://<your-domain>/auth/callback`)이 등록되어 있는지 확인하십시오.
8. **최종 보고 (Report)**: 
   - 모든 검증을 성공적으로 통과한 경우에만, 수정된 내용과 테스트 결과, 그리고 최종적으로 달성한 작업 내역을 사용자에게 정리하여 간결하게 보고합니다.
   - **보고서 하단에 "지침 준수 체크리스트 확인 결과" 및 "자가 점검 완료 리포트"를 포함합니다.**
## 3. 유용한 제어 명령어
- `/memory show`: 현재 적용 중인 모든 지침(Global + Project)을 확인합니다.
- `/memory refresh`: `GEMINI.md` 파일 수정 후 즉시 반영하고 싶을 때 사용합니다.
- `/init`: 프로젝트의 기술 스택을 자동 감지하여 기초적인 `GEMINI.md`를 생성합니다.

### 효율적인 협업 팁
- **단계별 접근**: 복잡한 작업은 Research -> Strategy -> Execution 단계를 명시적으로 거치도록 요청하세요.
- **피드백 루프**: 결과가 마음에 들지 않으면 지침을 `GEMINI.md`에 추가하여 다음번에 반영되도록 하세요.
- **문서 참조 (Critical)**: **모든 기능 개발 및 요구사항 분석 시 `docs/` 폴더 내의 기획 문서(예: `docs/00_요피랜드.md`)를 반드시 최우선으로 참조**하여 프로젝트의 목적과 정체성에 맞는 결과물을 도출해야 합니다.
